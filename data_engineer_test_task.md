# Data Engineer

---

## 1. Схема БД

Спроектировать схему БД для хранения пользовательских логов форума. Логи должны содержать следующие действия пользователя:

- первый заход на сайт
- регистрация
- логин / логаут
- создание темы
- заход на тему
- удаление темы
- написание сообщения

**Требования:**

- **1.1** Тему может создавать только залогиненный пользователь
- **1.2** Сообщение может писать любой пользователь — залогиненный или аноним
- **1.3** В логах должна храниться информация о пользователе, совершившем действие; описание действия; идентификатор объекта действия (например, id темы или комментария); ответ от сервера (ошибка или успех); дата и время действия
- **1.4** Таблица логов должна быть **партиционирована**. В README обоснуйте выбор стратегии партиционирования
- **1.5** Добавить обоснованные индексы. Выбор индексов описать в README

---

## 2. Инфраструктура (Docker)

Поднять окружение через **docker-compose**, которое включает:

- **PostgreSQL** — основная база данных
- **Apache Airflow** (версия 2.x) — оркестрация пайплайнов

Все сервисы должны подниматься одной командой:

```bash
docker-compose up
```

## 3. DAG: Генерация данных (`forum_data_generation`)

Написать Airflow DAG, который генерирует и загружает данные в БД.

### Требования к данным

- Датасет покрывает **один календарный месяц**
- Минимум **по 5 действий каждого типа** в день
- Для действия *создания темы* — минимум **2 случая ошибки** по причине отсутствия логина
- Для действия *написания сообщения* — примерно равное распределение залогиненных и анонимов
- Количество действий каждого типа выбирается **рандомайзером** с учётом минимальных порогов

### Структура DAG

```
generate_users
      │
      ▼
generate_logs        ←── рандомайзер с учётом минимальных порогов
      │
      ▼
validate_generated_data   ←── проверки: нет логаута без логина,
      │                        нет создания тем у незалогиненных
      ▼
load_to_postgres
```

## 4. DAG: Агрегация данных (`forum_log_aggregation`)

Написать Airflow DAG, который читает данные из БД и формирует итоговый `.csv` файл.

**Расписание:** DAG должен запускаться **ежедневно** и обрабатывать данные за предыдущий день. Также должна быть возможность запустить DAG вручную, передав произвольный период через **Airflow Params** (формат на ваш выбор).

### Итоговый датасет

| Поле | Описание |
|---|---|
| `day` | Дата |
| `new_accounts` | Количество новых зарегистрированных аккаунтов |
| `anon_messages_pct` | Доля сообщений от анонимов (% от всех сообщений за день) |
| `total_messages` | Общее количество сообщений за день |
| `topics_growth_pct` | Процентное изменение количества тем относительно предыдущего дня |
| `avg_session_duration_sec` | Средняя длина сессии в секундах (время между логином и логаутом) |
| `d1_retention_pct` | Retention D1: доля пользователей, вернувшихся на следующий день после регистрации |

### Структура DAG

```
check_data_availability    ←── Sensor: данные за период есть в БД
          │
          ▼
   extract_raw_logs
          │
          ▼
   validate_data           ←── проверка качества: нет неожиданных NULL,
          │                     значения в допустимых диапазонах
          ▼
  transform_aggregates
          │
          ▼
    export_to_csv
          │
          ▼
  alert_on_anomaly         ←── логирует предупреждение, если показатели
                                выходят за пределы нормы
                                (например, anon_messages_pct > 80%)
```

---

## 5. Качество данных

Для шагов `validate_generated_data` и `validate_data` использовать библиотеку **Great Expectations** или **pandera**. Валидация должна включать:

- Отсутствие неожиданных `NULL` в обязательных полях
- Корректность последовательности событий (нет логаута без логина, нет создания темы без авторизации)
- Числовые метрики находятся в допустимых диапазонах
